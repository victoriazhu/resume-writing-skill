# 简历写作 Skill（互联网行业通用版）

面向互联网行业多个序列的通用简历写作辅助（产品/技术/设计/运营/市场/数据/职能等）。根据用户选择的岗位类型，自动匹配该序列的**能力焦点、动词池、篇幅控制、模板风格和验证清单**，支持3种输入模式，从零到一生成可投递版本的简历。

---

> 用户在 Q1 回答岗位类型后，优先匹配已有模板。如果用户说的岗位**不在已支持的列表中**，系统应主动搜索（使用 web_search 或 duckduckgo-search）该岗位的核心能力要求、典型工作产出、简历常见动词、量化指标等，自动生成一个临时模板。

### 动态扩展：未知岗位的处理流程

当用户 Q1 说的岗位不在以上 12 个序列中时：

1. **搜索动作**：使用 `web_search` 或 `duckduckgo-search` 搜索以下 3 个方向：
   - `[岗位名称] 岗位职责 核心能力 要求`
   - `[岗位名称] 简历 关键词  HR筛选`
   - `[岗位名称] 量化指标 KPI`
   > 注意：搜索不要问用户"要我搜吗"，直接搜。搜完后也不要向用户汇报搜索过程，直接基于搜索结果生成模板。

2. **模板生成**：根据搜索结果，提炼出该岗位的：
   - `core_skills`（3-5 个核心能力关键词）
   - `verb_pool`（该岗位简历中常用的有力动词）
   - `quant_tendency`（该岗位常见的量化指标类型）
   - `must_have`（该岗位简历中必须具备的要素）

3. **模板缓存**：生成的模板仅在本次对话中生效。不写入 skill 文件。

4. **学习反馈**：对话结束后，问用户"刚才生成的[岗位名称]模板是否准确？要不要我更新到简历写作技能中，以后其他人用也能自动识别？"如果用户同意，用 `skill_manage(action='patch')` 将该岗位追加到已有模板表中。

### 设计原则
- 每个岗位的量化倾向不同——后端写QPS和SLA，设计写NPS和组件复用率，财务写成本节省率，不要套用PM的数据模板
- 动词池也不一样——后端多写"设计、实现、重构"，设计多写"定义、统一"，运营多写"策划、执行"
- 核心原则：**让HR一眼在第一行就知道这个候选人是什么岗位、什么级别**

### 跨岗位共性的通用要求

不论什么岗位，以下为强制要求：
1. **每条要点含量化结果结尾**
2. **动词有力不重复**，匹配岗位动词池
3. **时间线倒序**
4. **无第一人称**
5. **突出角色定位**（第1条统领定调）

## 三种工作模式

本Skill根据用户输入自动判断模式，无需用户手动选择。

### 模式一：从0开始生成（默认模式）

**触发条件**：用户没有提供任何已有简历或文档素材，直接说"帮我写简历"、"生成简历"。

**流程**：执行完整9问采访流程（Q1-Q9），逐步采集信息后生成完整简历。

### 模式二：基于现有简历优化

**触发条件**：用户提供了现有简历文本或文件路径，要求"优化简历"、"改简历"、"升级简历"。

**执行流程**：
1. 读取用户提供的原简历文件（支持PDF/Markdown/纯文本，PDF使用PyMuPDF提取）
2. **自动提取结构信息**：个人信息、时间线与公司（按倒序排列）、工作经历要点（识别弱动词）、项目经历（区别"项目"和"日常职责"）、个人优势、教育背景
3. **诊断原简历问题**：经历项目是否混排、弱动词比例（>30%需全部改写）、量化数据缺失、约束条件缺失、时间线是否倒序、核心优势是否有佐证
4. 仅追问缺失或模糊的部分
5. 基于原简历结构+补充信息，按Skill模板重新组织输出

**核心原则**：
- 保留原简历中已确认的准确信息
- **已有信息足以输出时就先输出优化版**，附改动说明，再问用户是否需要迭代
- 输出时标注改动点（弱动词改写、经历/项目拆分、时间线重排、新增量化等）

**追问示例**：
- "你原简历中写'负责提升用户体验'，能具体说一下提升了什么指标？提升了多少？"
- "你提到'推动项目上线'，这个项目最终带来了什么业务结果？有量化数据吗？"

### 模式三：从项目文档/工作文档提炼

**触发条件**：用户提供了项目文档、工作总结、OKR复盘、述职报告等素材，要求"从这些文档提炼简历"。

**执行流程**：
1. 读取用户提供的所有文档
2. 扫描文档内容，识别和提取关键信息（公司名、时间段、title、负责模块、指标数据、项目背景、关键动作、业务结果等）
3. 将提取的信息填入9问框架
4. 对缺失信息进行定向追问（只追问文档中没有覆盖到的部分）
5. 组装为完整简历草稿输出

**核心原则**：
- **保持原文事实不变**：不做推理、不补充文档中没有的数据
- **组织方式可调整**：按简历逻辑重新排列信息，突出业务结果
- **注明信息来源**：对从文档中提取的每条关键信息，标注来源文档名称，方便用户回溯验证

**追问示例**：
- "从你提供的工作总结中我提取了X、Y、Z这几个项目，还有别的项目要补充吗？"
- "项目A你写了'最终提升了转化率'，但文档里没写具体数值，方便补充吗？"

### 模式选择速查表

| 用户输入 | 匹配模式 | 执行要点 |
|---------|---------|---------|
| "帮我写简历" / "生成简历" | 模式一：从0开始 | 完整9问采访 |
| "改简历" / "优化简历" + 发来文件 | 模式二：优化 | 提取+追问缺失 |
| "看看我的简历" + 链接/路径 | 模式二：优化 | 自动提取结构 |
| "从这些文档提炼简历" + 项目文档/OKR/总结 | 模式三：提炼 | 扫描→提取→追问缺失 |
| 发来多个文档说"帮我写简历" | 模式三：提炼 | 全量扫描提取 |
| 发来简历+项目文档说"帮我优化" | 模式二+三混合 | 以简历为骨架，文档补充细节 |

## 触发条件

用户说以下任意一句时加载本技能：
- **模式一**："帮我写简历" / "生成简历" / "写一份简历"
- **模式二**："改简历" / "优化简历" / "升级简历" / "简历美化" + 提供了现有简历
- **模式三**："从这些文档提炼简历" / "帮我根据项目文档写简历" + 提供了项目文档/总结/OKR
- 或在求职辅导项目语境下提及"简历"（自动判断模式）

### 核心能力分层

#### 校招层(0-3年)
- **定位**：潜力展示型，学历>实习>项目>课程
- **强调**：学习力、执行力、基本功(逻辑/数据/文档)、产品sense
- **动词**：参与、协助、支持、执行、落地、完成、输出、分析
- **不写**：不写"负责XX"、不夸大角色
- **加分**：竞品分析报告、原型、调研、数据分析作品

#### 中阶层(3-8年, P6-P7/2-1~2-2)
- **定位**：执行能力型，独立负责模块，拆解目标拿结果
- **强调**：独立负责、数据驱动、跨部门推动、专业深度
- **动词**：推动、主导、搭建、设计、定义、优化、建立
- **不写**：不写"带领团队"(除非实线带人)
- **差异**：P6写模块闭环，P7写跨模块影响力

#### 高阶/专家层(5-12年, P7-P8/3-1)
- **定位**：策略+管理型，定义方向、搭体系、带人拿结果
- **强调**：策略规划、体系搭建、团队管理、方法论沉淀
- **动词**：带领、制定、重构、建立、驱动、定义、统筹
- **必写**：约束条件(成本/资源红线)、多端利益平衡、可复用方法论
- **差异**：P7写执行闭环+数据，P8写策略设计+体系+团队梯队

#### 资深总监层(10年+, P9+/3-2+)
- **定位**：战略型，做判断、定方向、建组织、推动变革
- **强调**：全局观、0-1能力、组织建设、行业影响力、商业结果
- **动词**：规划、定义、制定、带领、统筹、决策
- **必写**：团队规模、跨BG协同、行业影响力事件

## 核心原则

### 篇幅控制(硬性约束)

> **所有级别统一控制：无论初阶、中阶、高阶，简历总篇幅不得超过 2 页 A4。** 初阶应控制在 1 页内，中阶 1-1.5 页，高阶最多 2 页。

| 岗位级别 | 简历篇幅上限 | 工作经历 | 项目经历（个数不限） |
|---------|------------|---------|------------------|
| 初阶(专员/助理/校招0-3年) | **≤1页** | ≤2段+实习，每段2-3条要点 | 选1-3个，每个2-3条难点 |
| 中阶(经理/高级/主管3-8年) | **≤1.5页** | ≤3段，每段3-4条要点 | 选核心的2-4个，每个2-3条难点 |
| 高阶(总监/专家/负责人8年+) | **≤2页** | ≤4段，每段3-5条要点 | 不限个数，但**每个≤4条难点，每条≤2行** |

**超限处理**：生成完成后自动估算篇幅，超限时自动裁剪：
- 保留最核心的，删除重复性或次要内容
- 次要项目合并为一句话，不单独展开
- 每条要点控制在 ≤2 行（约 40-60 字），超长则精简
- 裁剪后如果仍然超限，删除非核心的工作经历段落

### 内容质量（核心：首版即交付）

**首版生成的简历质量即决定了用户体验和口碑传播**。因此：

- **每条要点必须以量化结果或明确的业务贡献结尾** — 这是简历的硬通货，没有数据的经历等于没写
- 禁止模糊表述("负责"/"参与"/"协助"必须转化为具体动作+结果)
- 经历以**业务结果贡献**结尾，不是以"负责XX"结尾
- 项目经历要体现：背景→角色→行动→结果(隐含STAR)

### 核心优势撰写原则
- **每条优势必须附带具体佐证**（数据/项目/场景），不能写空洞的"擅长XX"
- 高阶岗位的优势要有**可感知的复杂度支撑**（如"能在强约束下达成多目标"而非"有复杂项目管理经验"）
- 优势应体现**差异化标签**，而非通用能力（如"治理体系1号位经验"优于"良好的沟通能力"）

### 语言风格
- 专业、简洁、数据驱动
- 使用动词开头(主导/搭建/重构/推动/建立/优化/定义等)
- 禁止第一人称(我/我们)，用隐式主语
- **动词要与动作规模匹配**：带队用"带领/统筹"，个人执行用"推动/主导"，策略设计用"定义/设计"

### 篇幅与节奏控制
- 高阶岗位的工作经历可控制在4-5条要点/段，但**每段必须有1条统领性要点**（用"为XX负责"开头定位角色）
- 跨公司经历的**时间线要干净**：每段经历标注开始-结束年月，重复公司合并连续时间

> - ✅ 要写"重建XX模块后，效率提升40%，人力节省60%"（这是成果，HR看了觉得你是个能解决问题的人）
> - ❌ 不要写"参与XX系统的开发和上线"（谁都能写）
> - ✅ 要写"主导XX系统的架构重构，系统QPS提升3倍，硬件成本降低30%"（有数字、有结果、有说服力）
> - **每条要点必须是「动词 + 你做了什么 + 可量化的结果」的三段结构，缺一不可**

```
**[公司名] | [title]（[角色定位，如：XX领域1号位]） | [时间段]**
• **为[核心指标]负责**：[统领性要点，定位你在该段的角色和权责范围]
• [动词] + [业务动作] + [量化过程指标] → [直接业务结果]
• [动词] + [业务动作] + [量化过程指标] → [直接业务结果]
• [动词] + [业务动作] + [量化过程指标] → [直接业务结果]，同时满足[约束条件]
```
- 第1条要点用**粗体`为XX负责`**统领定调，让HR一眼看清这段经历的核心价值
- 后续要点遵循**动词→业务动作→量化指标→业务结果**的四段结构
- 高阶岗位的要点中如果有约束条件（成本红线/资源限制），一定要写出来，**约束越强结果的含金量越高**
- **跨段的同一家公司**要在title处区分角色演进（如"产品专家→高级产品经理"）

**中阶/初阶模板**：
```
**[公司名] | [title] | [时间段]**
• [动词] + [业务动作] + [量化指标] → [业务结果]
• [动词] + [业务动作] + [量化指标] → [业务结果]
• [动词] + [业务动作] + [量化指标] → [业务结果]
```

动词池参考：定义/制定/搭建/重构/优化/建立/推动/主导/落地/实施/交付/上线/提升/拉动/驱动/转化/扩展/带领/管理/协调/统筹

### Step 3b: JD 匹配与定向改写（可选）

> **仅在用户提供了目标 JD 时执行**。如果用户没有提供 JD，跳过此步骤，直接到 Step 4。

**流程**：

1. **用户输入 JD**：用户粘贴目标岗位 JD 文本（或 Agent 在 Q5-Q6 后主动问"你有目标 JD 吗？贴给我，我帮你匹配关键词"）。
2. **关键词提取**：从 JD 文本中提取核心关键词（通常 5-10 个），包括：
   - 硬技能关键词（技术栈/工具/方法）
   - 软技能关键词（沟通/推动/组织）
   - 领域关键词（电商/AI/金融等）
   - 经验关键词（5年以上/从0到1/团队管理）
3. **匹配度分析**：对比关键词与已生成简历的覆盖情况。输出：
   - 匹配度评分（百分比）
   - 已匹配关键词列表
   - 待补充关键词列表
   - 核心差距分析
4. **定向改写**：针对待补充关键词，自然嵌入简历现有的工作经历和项目经历要点中。改写原则：
   - 不编造经历——用户没有做过的事不写
   - 可重新措辞——用户做过但没用这个关键词时，用更贴近 JD 的表述
   - 每个嵌入点标注"JD 优化"标记

**输出示例**：
```
匹配度：72%
✅ 已匹配（6/8）：AI产品、产品规划、跨部门协调、大模型、数据分析、产品路线图
⚠️ 待补充（2/8）：SQL、AI应用落地

👉 建议改写工作经历第2条：将"通过数据驱动产品优化"改为"通过SQL提取用户行为数据驱动产品优化"
👉 建议在核心优势中突出"AI+跨境电商的跨界能力"
```

**核心写作理念（必须先读）**：

简历的项目经历不是技术文档、不是立项报告、不是复盘总结。HR 读你的项目经历不是为了理解"这个项目做了什么"，而是为了判断**"这个人能力有多强"**。

**推荐结构：「难点 → 行动 → 结果」**：

> 不限制具体结构（你也可以用三段结构或其他方式），但**每个项目写清楚三个问题**：
>
> 1. **难点是什么？** —— 这个项目最难的地方在哪里？技术瓶颈？业务复杂？组织阻力？时间压力？资源限制？
> 2. **你做了什么关键行动？** —— 针对这个难点，你做了什么决定/方案/动作？
> 3. **拿到了什么结果？** —— 可量化的最终成果是什么？

**所以**：
- ✅ 可以写背景/难点（有助于理解为什么"难"）
- ✅ 可以写具体做法（证明你的能力）
- ✅ **但不要把难点写成立项报告**——不是"根据业务需求决定做XX系统"，而是"当系统QPS达到峰值10万时，核心链路延迟暴增"
- ❌ **不要只写做了什么而没写结果**——行动没有结果就没有说服力
- ❌ **不要只写结果没写难点**——结果没有被"难"衬托，HR觉得谁都能做到
- ✅ **最佳结构：先抛出让人"哇"的难点，再亮出你的解法，最后用数据证明结果**

> 一句话指导原则：**HR 要的是"这个人有多厉害"的证据，不是"这个项目怎么做"的过程。** 而"这个人有多厉害"最有力的证据 = 难以解决的问题 × 你特有的解法 × 可量化的产出。

每个项目格式：

**高阶岗位项目模板**：
```
### [项目名称] | [时间段]

**核心能力标签**：[策略设计/体系搭建/技术驱动/组织推动/数据洞察/多端平衡/从0到1等]
> 标签对应当前 `role_type` 的能力语言。非PM岗位不要套用PM的标签。

**【难点与关键行动】**
[难点1] → [你的解法] → [可量化结果]
[难点2] → [你的解法] → [可量化结果]
[难点3] → [你的解法] → [可量化结果]

**【AI技术】（如需）**
[LLM语义理解 / 多模态视觉 / 时序预测 / Agent编排等]

**【能力沉淀】（高阶必填）**
[通过这个项目沉淀的方法论/可复用的体系资产/行业认知突破]
```
- **核心能力标签**必须显式标注，让HR一眼看到这个项目展现了你的什么能力
- **每条「难点→解法→结果」限制在 1-2 行以内**（约 30-50 字），禁止段落式描述。HR 扫读简历时每条只有 3 秒钟注意力
- ✅ 好的写法："QPS峰值10万时核心链路延迟暴增 → 引入多级缓存+异步写 → P99延迟从500ms降至80ms"
- ❌ 差的写法："随着业务快速发展，系统面临越来越大的并发压力，特别是在大促期间QPS峰值达到10万时，我们发现核心链路的响应时间出现了明显的上升趋势..."
- 每条之间不要写成流水账，要让人感觉"这件事的难度逐渐升级"
- 高阶岗位必须沉淀AI/技术应用经验（如有），增加技术领导力标签
- **能力沉淀**控制在 2-3 句话以内，同样禁止展开
- **每个项目总长度 ≤ 15 行**，超出自动裁剪

**中阶/初阶项目模板**：
```
### [项目名称] | [时间段]

**核心能力标签**：[遵循当前role_type的能力语言]

**【难点与关键行动】**
[难点1] → [你的解法] → [可量化结果]
[难点2] → [你的解法] → [可量化结果]

**核心产出**：[最终结果]
```
- **与高阶模板统一「难点→解法→结果」结构**，但篇幅更精简
- 不需要「能力沉淀」段落，但中阶用户可要求加入
- 2-3 条难点足够，聚焦最核心的挑战

### Step 5: 导出简历文件

简历生成后，按以下格式保存交付到用户指定的目录（如 `~/Desktop/`）：

1. **Markdown 源文件**（必存）：`V{version}_{岗位方向}.md`
2. **PDF 副本**（可选）：如有 pandoc 或 weasyprint，转换一份 PDF
3. **Word (.docx)**（可选）：如有 pandoc，转换一份 docx
4. **纯文本 (.txt)**（可选）：去除 markdown 格式符号

> 如果没有 pandoc/weasyprint，至少保证 markdown 格式。

命名格式：`V{version}_{岗位方向}.{md/pdf/docx/txt}`

### Step 6: 篇幅检查与重复内容处理

**强制检查**：生成完成后自动估算篇幅，超限必须裁剪。

- 估算标准：中文字符 + 英文单词混合计算，约 400-500 字/A4 页
- **超限即裁剪**，不询问用户——篇幅控制是 skill 的责任
- 裁剪优先级：①删除重复性内容 ②次要项目合并为一句话 ③压缩每条要点字数 ④删除非核心经历段落
- **所有要点单行不超过 2 行（约 40-60 字）**，超长必须拆分或精简

**重复内容检测与处理**：

> 如果检测到简历中存在大量重复内容（如多条经历描述相似、多个项目使用相同模板句式、不同时间段的职责雷同），按以下优先级处理：

1. **尝试获取更多信息**：主动引导用户提供项目文档或个人初版简历（"我注意到有几段经历的描述比较相似，如果你有更详细的项目文档或原始简历，发给我可以帮助写出更丰富的内容"）
2. **如果用户无法提供**：通过采访补充——针对重复的部分追问细节差异（"你在A公司和B公司的用户运营工作，策略上有什么不同？面向的用户群体一样吗？"）
3. **如果既无文档也采访不出差异**：对重复部分进行**重新编排**——合并相似经历，每段只保留最有代表性的 1 条，其他改为"同时负责XX"一句话带过，避免 HR 看到两三段长得很像的经历觉得你没有成长

## 避坑指南

1. **不要一次问完所有问题**：逐个问，答一个再问下一个
2. **不要编造数据**：用户给不出具体数据时用「XX%」占位标记高亮
3. **Q1-Q6收完立刻生成工作经历+核心优势**，不等项目经历
4. **每个项目单独详细展开**，不要合并到工作经历里
5. **高阶侧重策略+影响力+方法论**，中阶侧重执行+专业深度+独立负责。注意**序列差异**：产品经理写DAU/GMV/NPS，后端写QPS/SLA/系统规模，设计写NPS/组件复用率/设计系统，运营写活动ROI/留存，数据写指标覆盖率/决策采纳率，HRBP写招聘周期/离职率/组织满意度，财务写成本节省率/审计覆盖率，市场写CAC/ROI，项目管理写准时交付率/预算偏差率——不要串用其他序列的框架和数据模板
6. **积极追问**：用户描述不具体时追问数据
7. **平衡篇幅**：材料多时要主动筛选压缩
8. **模式二不推翻原简历**：保留用户已确认的信息，只增补和优化不重写
9. **模式三不编造文档外信息**：严格基于文档内容提取，缺失部分标记由用户补充
10. **拆分经历与项目**：原简历常见的"大段混排"问题（工作经历要点和项目经历混在同一个bullet list里），必须先拆分。日常职责放工作经历，有完整背景需放项目经历。
11. **已有信息优先输出**：模式二下，原简历提供的信息足够支撑优化版时就直接输出，不需要先追问。输出后附改动说明，用户不满意再迭代。
12. **时间线倒序是强制要求**：输出时严格按倒序排列（最近的工作在最前）。
13. **跨行业经验处理**：多公司经历且岗位不完全一致时，不删除"不相关"经历，缩短篇幅弱化处理，保留职业发展完整性同时突出目标岗位相关经历。

### 边界场景
- 用户只有1段工作经历：深挖写透，允许5-6条要点
- 校招：侧重项目经历(实习/课程项目)，工作经历换实习经历
- 跨行跳槽：突出可迁移能力，弱化行业背景
- 用户提供大量细节：筛选最突出的30%
- **用户同时提供简历+文档**：模式二+三混合，以简历为骨架用文档补充细节
- **文档过多（5个以上）**：先扫描目录/标题筛选最相关项目，再逐篇深入

### 简历投递策略建议

> **注意**：以下策略仅适用于 PM/运营/设计等对行业经验敏感的岗位。后端/数据分析/HRBP 等岗位的投递策略更依赖技术栈匹配和项目相关性，不在此模板中展开。
